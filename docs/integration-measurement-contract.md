# 計測コントラクト仕様(操縦席 要素4・要素5の青写真)

> 作成: 2026-05-30 / 親: [integration-roadmap.md](./integration-roadmap.md)
> 目的: 全サイロが吐く「横並び可能な1指標」と、それを食う再配分ルールを正式定義する。
> これが Phase 1(設計のみ・コード無し)の完成物。

## 0. 設計原則(調査から確定した3つ)

1. **共通の出力単位は1つに揃える。推計方法はサイロごとに変える。**
   crypto は5年14-15約定と希少で、株式回転と同じ「実現往復EV」では永久に測れない。
2. **「指標(τ)」と「証拠グレード」は分離する。** τ=最良の期待値推計、グレード=その確信度。
   混ぜると情報が潰れる。資本を動かす許可はグレードがゲートする。
3. **"ベータの受け皿"と"アルファの賭け"を区別する。**
   NISAインデックスは超過収益を主張しない=賭けでない=常時資金可。
   crypto/特定回転は超過収益を主張する=証拠でゲートする。

## 1. 共通指標 τ(タウ)= 税後・年率・資本あたり期待リターン

```
τ = (グロス年率期待リターン) × (税歩留まり係数)
```

- **年率**にするのは時計を揃えるため(+100%/2年 と +5%/20日 を比較可能に)。
- **資本あたり**にするのはサイズを揃えるため。
- τ とは別に **証拠グレード G1/G2/G3** を必ずタグ付けする。

## 2. 税歩留まり係数(山田家の実所得で再計算 ★ブリーフの数値を修正)

ブリーフの「crypto 45〜67%」は課税所得4000万超の高所得者前提。山田さんの実態:

```
給与額面 550万 − 給与所得控除 ~164万 = 給与所得 ~386万
  − 社会保険料 ~82万 − 基礎控除48万 等 ≈ 課税所得 ~256万(10%ブラケット)
```

→ crypto 雑所得は給与の上に積み上がり、課税所得256万から上のブラケットで課税:

| 区分 | 税率(所得税+住民税10%) | 税歩留まり | 備考 |
|---|---|---|---|
| **NISA** | 0% | **100%** | 母艦・複利装置 |
| **特定(株)** | 20.315%(申告分離) | **~80%** | 損益通算・3年繰越可 |
| **crypto** 〜年74万の益 | 20%(10%+10%) | **~80%** | 課税所得330万まで |
| **crypto** 〜年440万の益 | 30%(20%+10%) | **~70%** | 330〜695万帯 |
| **crypto** 695万超の益 | 33%(23%+10%) | **~67%** | 大相場の年のみ |

**重要な含意:** 山田家では crypto と特定の税差は小さい(80% vs 70-80%)。
ブリーフの「crypto=課税の重い稼ぎ頭」は**この所得帯では弱い**。
crypto の本当の不利は税率でなく ①株と損益通算できない ②益が塊で来て高ブラケットに跳ねうる、の2点。
NISA の 100% が両者を圧倒する構図は不変。
※ 社保・控除は概算。正確な課税所得は源泉徴収票で要確認(τ の精度に効く)。

## 3. サイロ別の τ 推計方法

### 株アクティブ(特定・20日回転)
- ソース: `realized_trades`(realized_gain は手数料込み実額)。
- グロス年率 ≈ (1 + r̄)^(365/h̄) − 1 × 稼働率
  - r̄ = 平均実現リターン(realized_gain / 取得簿価)、h̄ = 平均保有日数、稼働率 = 資本が建玉に入っている時間割合(待機ドラッグを引く)
- τ = グロス年率 × 0.80
- 先行指標(実現が溜まる前): `exit_judgments` の pct_at_horizon × decision_was_right 率 を擬似グロスとして同式で年率化。

### crypto(暴落買い・年数回)
- 実現往復は待てない。3階建てで推計:
  - グロス年率 = バックテストCAGR(30-35%) × min(1, ライブ追従係数)
  - ライブ追従係数 = `monthly_snapshots.cagr_running` ÷ バックテストCAGR(12ヶ月未満は未確定)
  - 補助シグナル: 含み建玉が +100% へ向かうか −60% へ滲むか(低グレード)
- τ = グロス年率 × 0.70(益が中規模で30%帯に入る前提・保守側)
- **操縦席は crypto を測り直さない。** crypto 自前の `monthly_snapshots` を税後・年率に翻訳するだけ。

### NISAコア(オルカン等・買い持ち)
- 取引でない。グロス年率 = 期待市場リターン ~5〜7%。
- τ = 5〜7% × 1.00。**ベータの受け皿**として常時資金可(後述)。

## 4. 証拠グレードと卒業閾値(★訂正: バックテスト実証を正当に格上げ)

**重要な訂正:** バックテスト・エビデンス(過去データの walk-forward・過去時点判断の反復実証)と、
ライブ実現エビデンス(稼働後の実約定)は**別の軸**。前者は厚く存在し、戦略を稼働させる正当な根拠。
後者だけがゼロ(稼働10日)。両者を潰してはならない。バックテストが閉じられない唯一の隙間=
**ライブ追従ギャップ**(スリッページ・手数料・GMO制約・レジーム変化・実装バグ)。これだけが残る未知。

| グレード | 意味 | 資本の扱い |
|---|---|---|
| **G3 ライブ実証** | ライブがバックテストを許容内で追従(十分な期間/N) | **設計サイズを超えた増資**を正当化(部屋530内) |
| **G2 バックテスト実証** | walk-forward で実証済。ライブ未確認 | **設計サイズでの稼働**を正当化(賭けではない・既定路線) |
| **G1 理論値** | 市場期待のみ(NISAベータ等)。評価ゲート対象外 | ベータ受け皿として常時可 |

**現在地: crypto も株アクティブも G2(バックテスト実証済)。** → 設計サイズでの稼働は正当。
- crypto は既に設計サイズ(BTC60+アルト140=200万)で稼働中 = G2根拠で適正。凍結ではなく適正サイズ。
- 株アクティブは walk-forward で5セグメント実証済(segment.ts)だが特定口座が15万=**設計サイズ未満**。
  → 設計サイズまで埋めるのは「賭け」でなく「実証済戦略を設計どおり配備する」。

卒業閾値 G2→G3(=設計サイズを超える増資の解禁):
- **crypto**: ライブ12ヶ月以上 & `cagr_running` ≥ +20%(**crypto自前ROADMAP Phase 2ゲートと一致**)。
  実現往復は希少なので時価ベース cagr_running で追従を測る。
- **株アクティブ**: バグ修正(2026-05-30: 保有日数・AI失敗時売却)後の**前向き**実現が
  バックテスト期待値に追従(実現N≥30 かつ 95%CI下限>0)。修正前のライブは実装≠BTで無効。
- **NISAコア**: 市場ベータ・超過収益を主張せず=恒久 G1扱い・評価ゲート対象外(常時可)。

## 5. 再配分ルール骨格(Phase 4・τ と グレードを食う)

```
入力: 各サイロの (τ, グレード, 現配分, キャップ)

1. ハード制約を先に満たす(不可侵):
   - 総資産が床980万を割らない
   - crypto ≤ 部屋530万、deployed risk(crypto+特定)≤ 部屋530万
   - 生活防衛400 + 教育200 はロック
2. 既定の受け皿 = NISAベータ複利:
   G3アルファに正当化されない自由資本は、すべてNISAインデックスへ流す
   (年間枠・夫婦2口座・夫→妻の贈与110万線を尊重)
3. アルファ増資:
   サイロが G3 かつ τ が NISA受け皿のτ を「リスクプレミアム分」上回る時のみ
   部屋530の範囲で増資。crypto の高分散・塊課税・低流動性を踏まえ、
   NISAのτ(5-7%)を 2〜3倍 上回ること(=τ≥15%)を目安。
   ※ crypto BT後 τ≈22-24% は条件を満たすが、G3到達まで12ヶ月は増資しない
4. それまで = 🛡️保全モード: 現サイズ凍結、自由現金はNISA受け皿へ
```

## 5.5 第2の最適化軸:実現時間(time-to-realization)

操縦席は τ(リターン)だけでなく**速さ**も最適化する(ユーザー方針: 費用対効果・実現時間短縮・最適路算出)。
増資の絶対額は**前もって固定しない**。再配分ルールは絶対額を自動執行せず、
「G3到達・τ・部屋の余地・費用対効果・実現時間への影響」を提示し、**引き金はユーザーが引く**(相談ゲート)。

**情報コストとしての検証予算(exploration vs exploitation):**
- 再配分マシンは「最初の G3」が出るまで保全モードのまま動かない。
- → **最速で何かを G3 に到達させること自体が全体の実現時間を縮める。**
- crypto は実証まで12ヶ月(時計が遅い)。株アクティブ(20日回転)は数ヶ月でN≥30(最速)。なのに現在ほぼ空(15万)。
- → 株アクティブに小さな**検証予算(例 50〜100万)**を置くのは「賭け」でなく
  「証拠を最速で買う情報コスト」。検証サイズ厳守。最速の時計を回し再配分マシンの起動時間を短縮する。
- ※ これも絶対額決定なので引き金はユーザー。τ評価が先行指標(exit_judgments)で正に出ているのが前提。

## 5.7 戦略の畳み込み(2026-05-30 ユーザー判断)

**方針: 「crypto がアルファを担うなら、株は全部インデックスでよい」。** 3層→2層へ簡素化:
```
旧: コア(NISA index) / アルファ(特定×AI回転) / サテライト(crypto)
新: ベータ複利(NISA index全集約) / アルファ(crypto 1本)
```
- 理由: アルファエンジン2本は不要。crypto は全自動・哲学(機械的AI運用)に最適合・G2実証済。
  特定回転は複雑・人介在・税20%。crypto がアルファを担えるなら第2エンジンは冗長。
- 失うもの: 非相関アルファの分散 / 最速の証拠時計(20日回転)。→ 畳み方は段階的(下記)。

**特定アクティブの設計サイズ = 当面ゼロ(株は全インデックス)。**
- 特定は「AIの購入判断・利確判断の向上待ち」(ユーザー方針)。
- **証拠は特定に資本を晒さず測れる:** 購入判断=`prediction_signals`+signal track / 利確判断=`exit_judgments`の decision_was_right・pct_at_horizon。
- → 先行指標が「買いも売りも良い」と実証したら初めてBT根拠で特定を起こす。示さなければ恒久ゼロ=全インデックス確定。
- (旧 5.5 の「証拠を最速で買う検証予算」は撤回。弾を入れずに判断品質を測れる方が安い。)

## 5.8 出口品質スコアカード — Step 2 評価ジョブの設計(反実仮想)

Step 1 で `altcoin_exit_judgments` に判断を記録。Step 2 は各判断を後日採点し
`pct_at_horizon` / `decision_was_right` を埋める。価格源は `altcoin_price_history`
(6銘柄・日中粒度・2025-04〜)で任意の過去日を引ける。

**反実仮想の定義(=「今売る」を基準に、その判断が得だったか):**
- 判断時点の基準 = `unrealized_gain_pct`(その瞬間に売れば確定できた益率)。
- ホライズン時点の益率 `pct_at_horizon` = (price_at_horizon − buy_price) / buy_price × 100。
- **HOLD が正解** ⇔ `pct_at_horizon ≥ unrealized_gain_pct + 許容`(伸ばして得した)。
- **TAKE_PROFIT が正解** ⇔ `pct_at_horizon ≤ unrealized_gain_pct + 許容`(売却後に伸びず、利確が正当)。
- 許容(tolerance)= 数pp(ノイズ吸収。初期値 +5pp 程度を提案)。

**ホライズン:** crypto は「+85%で売る vs +150%まで伸ばす」の判断なので株の7/14日では短すぎる。
初期値 **30日**を提案(方向性は掴め、待ちすぎない)。将来は「実際にクローズした時点の実現益」での
確定評価も併用(最も正確だが時間がかかる)。`eval_horizon_days` に採用値を記録。

**配管:** crypto-bot に評価 cron(`/api/cron/exit-eval` 等)を新設。
`decision_was_right IS NULL AND judgment_at + horizon ≤ now` の行を拾い、
price_history から horizon 時点価格を引いて採点。株側の土曜 evaluate cron と同型。

**実装タイミング:** 現在ログは0件(アルトが+80%到達するまで評価対象が生まれない)。
器は急がず、定義をユーザーと固めてから、データが必要になる前に実装すればよい。

## 6. この仕様が確定させる「今の正解」(訂正後)

- crypto・株アクティブは **G2(バックテスト実証済)**。→ **設計サイズでの稼働は正当**(賭けではない)。
  - crypto は既に設計サイズ200万で稼働 = 適正。「凍結」ではなく「設計どおり」。
  - 株アクティブは設計サイズ未満(特定15万)。**設計サイズまで埋めるのはBT根拠で正当**(検証サイズの賭けではなく、実証済戦略の配備)。
- **設計サイズを超える増資**だけが G3(ライブ追従)待ち。これがあなたの「実証されたら増資」の正体。
- NISA は評価ゲート対象外の受け皿 → **自由現金200万・将来の実現益はNISAへ流すのが常に正解**。
- 操縦席は (τ, グレード) で機械的に動く。山田さんの「賭けには張らない、証明されたら寄せる」=
  「**設計サイズまではBT根拠で配備、それを超える増資はライブ実証ゲート**」とルール化される。
